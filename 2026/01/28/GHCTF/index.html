任意文件读取

  1. 漏洞的本质是：Web应用在处理文件路径时，没有对用户输行严格的校验和过滤，导致攻击者可以通过构造特殊路径（如 ../ ）来突破应用的访问限制，访问到预期之外的文件。
    问题：os.path.join(“static”, filename) 未过滤../等路径跳转字符，导致攻击者可构造路径读取任意文件。若用户传入filename=../app.py，则实际路径为static/../app.py，即读取服务器根目录下的app.py文件。

  1. 典型利用方式
    (1) 读取系统敏感文件
    在CTF或渗透测试中，这是最常见的目标：
    Linux系统： /etc/passwd （用户列表）、 /etc/shadow （密码哈希）、 /proc/self/environ （进程环境变量）
    Windows系统： C:\Windows\system32\drivers\etc\hosts 、 C:\Windows\win.ini 
    Web配置： /var/www/html/config.php 、 /app/.env （环境变量文件）

目录穿越

字符 含义 作用
. 当前目录 表示“就在当前文件夹下”
.. 上级目录 表示“回到上一级文件夹”
./ 当前目录简写 ./1.txt = 1.txt
../ 上级目录简写 ../1.txt 表示上一级目录下的 1.txt

基础示例（Linux系统）：

  • 当前路径： /var/www/html/ 
  •  ./static  → 解析为  /var/www/html/static （当前目录下的static）
  •  ../config  → 解析为  /var/www/config （上一级目录下的config）
  •  ../../etc  → 解析为  var/etc （跳两级到根目录下的etc）

SSRF

一、SSRF 定义与核心原理
Server-Side Request Forgery（服务器端请求伪造） 是一种Web安全漏洞，攻击者可通过构造恶意请求，诱导服务器向非预期目标（如内网服务、本地系统）发起请求，从而窃取敏感数据、绕过访问控制或执行未授权操作。让目标服务器替我们访问它自己的内网接口（仅允许本地访问），执行命令读取/flag。
典型场景：服务器信任本地/内网请求（如管理后台仅允许127.0.0.1访问），攻击者利用SSRF让服务器“代打”，突破权限限制。
二. 常用函数

  • file_get_contents函数：将整个文件读入一个字符串
  • fsockopen函数：打开一个网络连接或者一个unix套接字连接，使用该函数可以实现用户指定url中的数据，这个函数会使用socket与服务器进行TCP连接，传输原始数据
    三、SSRF 绕过策略
  1. 绕过黑名单过滤
    IP混淆：用127.1（等价于127.0.0.1）、0（部分系统解析为本地IP）替代被屏蔽的127.0.0.1。
    域名欺骗：注册指向本地IP的域名（如spoofed.burpcollaborator.net）。
    短网址：sina.lt
    编码/大小写绕过：对关键词URL编码（如127.0.0.1→%31%32%37%2E%30%2E%30%2E%31）或大小写混合（如LocalHost）。
    数制转换
  2. 绕过白名单过滤
    @符号注入：构造https://白名单域名@恶意域名（如https://allowed.com@evil.com），服务器可能优先解析@后的恶意域名。

#片段混淆：用https://恶意域名#白名单域名（如https://evil.com#allowed.com），部分解析器会忽略#后的内容。
子域名嵌套：注册allowed.com.evil.com，利用DNS层级欺骗白名单验证。
3. 利用开放重定向
若服务器仅允许白名单域名，但该域名存在开放重定向漏洞（如/redirect?url=http://evil.com），可构造：
http://白名单域名/redirect?url=http://内网IP/admin
服务器会先验证白名单，再跟随重定向访问内网目标。
四、SSRF 支持的协议分析
SSRF不仅限于HTTP/HTTPS，还可能支持以下协议，扩大攻击面：
file://：读取服务器本地文件（如file:///etc/passwd）。
gopher:：Gopher协议是一种早期的互联网分布式文档传输协议，通过客户端-服务器模型提供层级化的资源访问服务，支持文件浏览、搜索查询等功能。它以简洁的文本交互为核心，曾是HTTP普及前的主流互联网协议之一。

SSTI注入利用

一、什么是 SSTI？
用户可控的数据，被直接塞进了模板引擎的渲染过程里，导致攻击者可以注入模板语法，执行任意代码。
它和 XSS 不一样：
XSS 是前端执行，危害有限
SSTI 是服务器后端执行，可以直接拿服务器权限
二、漏洞的根本成因（核心）

  1. 模板引擎的工作机制
    模板引擎（如 Jinja2、Twig、Smarty、Velocity 等）的作用：
    把 HTML + 变量 / 逻辑 渲染成最终页面。
    例如 Flask/Jinja2 正常写法：

这是安全的，因为：
变量是被动渲染
模板结构是固定的
用户输入只当 “纯数据”
2. 危险写法 = 漏洞的直接原因
开发者错误地把用户输入直接拼进模板字符串里，再让模板引擎去渲染。
不安全代码示例（Flask）：

用户输入 不再是普通变量
用户输入 变成了模板本身的一部分
模板引擎会解析用户输入里的模板语法
攻击者只要传入：?name=49服务器就会执行 7*7，返回：Hello 49
这就是 SSTI 漏洞。
三、SSTI 漏洞成因总结（最核心 3 点）

  • 用户输入可控
  • 用户可以完全控制传入的字符串。用户输入被直接拼接到模板字符串中不是作为变量传入，而是成为模板代码的一部分。
  • 后端模板引擎会解析执行模板语法
    只要同时满足这 3 条，100% 存在 SSTI 漏洞。
    四、安全写法 vs 危险写法（对比一看就懂）
    ❌ 危险（SSTI）
    python
    运行
    template = “Hello “ + user_input
    render_template_string(template)
    ✅ 安全
    python
    运行
    render_template_string(“Hello “, name=user_input)
    区别：
    危险：用户输入 → 模板代码
    安全：用户输入 → 变量值
    五、为什么 SSTI 能变成 RCE（远程代码执行）？

一旦执行，服务器直接被拿下。
核心前提：Jinja2 与 Python 上下文的关联
Jinja2 虽然是模板引擎，但它运行在 Python 解释器环境中，并且没有完全隔离 Python 的底层对象和方法—— 这是能实现 RCE 的根本。
正常情况下，Jinja2 只允许渲染变量、简单逻辑（如 if/for）；
但通过 Python 的魔法方法（Magic Methods），攻击者可以突破模板引擎的限制，直接调用 Python 内置函数、操作系统接口，最终执行任意命令。

PHP序列化和反序列化

  1. 序列化：复杂数据类型转化成字符串。serialize() 是 PHP 内置函数
  2. 反序列化：把序列化后的字符串还原成原来的 PHP 数据类型（数组、对象）。unserialize()
    序列化结果分析

  1. 数组序列化格式：a:长度:{键类型:键长度:”键值”;值类型:值长度:值;…}
    a:3：表示数组（array），长度为 3；
    s:2:”id”：字符串（string）类型的键，长度 2，值为 id；
    i:1001：整数（int）类型的值，值为 1001。

  2. 对象序列化格式：O:类名长度:”类名”:属性个数:{属性类型:属性长度:”属性名”;属性值;…}
    私有属性：序列化后会带上类名前缀（如Userage）；
    受保护属性：序列化后会带上前缀（如gender）。

PHP反序列化的识别和利用

可控的反序列化输入 + 存在危险魔术方法的类 = 反序列化漏洞。

魔术方法 触发时机 风险点
__destruct() 对象被销毁时（脚本结束 /unset） 执行文件写入、命令执行等
__wakeup() 反序列化时自动触发 执行数据库操作、系统命令
__unserialize() 反序列化时自动触发 如果类中同时定义了 __unserialize() 和 __wakeup() 两个魔术方法， 则只有 __unserialize() 方法会生效，__wakeup() 方法会被忽略。此特性自 PHP 7.4.0 起可用。
__toString() 对象被当作字符串使用时 拼接恶意字符串触发代码执行
__call() 调用不存在的方法时 伪造方法调用执行危险操作
__invoke() 对象被当作函数调用时 直接执行函数 / 命令
__get() 访问对象的不存在/不可访问属性时自动触发
__unset() 在不可访问的属性上使用unset()时触发

3.正则匹配函数

\d：匹配数字
\w：匹配字母 / 数字 / 下划线
^：行开头，$：行结尾
+：匹配 1 次或多次，*：匹配 0 次或多次
/i：忽略大小写，/g：全局匹配（PHP 中需循环实现）

(1) preg_match () - 匹配一次
功能：在字符串中匹配正则表达式，只匹配第一次出现的结果。

返回值：匹配成功返回 1，失败返回 0，出错返回 false。
参数：

  • $pattern：正则表达式（必须用分隔符包裹，如/pattern/）
  • $subject：要匹配的字符串
  • $matches：可选，存储匹配结果（$matches[0]是完整匹配，$matches[1]是第一个子组）
  • $flags:常用PREG_OFFSET_CAPTURE：存储匹配结果的起始位置（如$matches[0][1]是位置）
  • $offset:避免从头匹配，提升性能（比如已知前 100 个字符无匹配）
    (2) preg_match_all () - 全局匹配*
    功能：匹配字符串中所有符合正则的结果，而非仅第一次。
    语法：int preg_match_all(string $pattern, string $subject [, array &$matches [, int $flags = PREG_PATTERN_ORDER [, int $offset = 0 ]]])
    返回值：匹配到的结果数量，失败返回 false。
    (3) preg_replace () - 正则替换
    功能：按正则规则替换字符串中的内容。
    语法：mixed preg_replace(mixed $pattern, mixed $replacement, mixed $subject [, int $limit = -1 [, int &$count ]])
    参数：
    $limit：可选，最多替换次数（-1 表示无限制）
    $count：可选，存储实际替换的次数

(4) preg_split () - 正则分割
功能：按正则规则分割字符串为数组。
语法：array preg_split(string $pattern, string $subject [, int $limit = -1 [, int $flags = 0 ]])
示例：分割不规则的分隔符
